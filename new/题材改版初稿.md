# 项目代号：辐射废土——修仙游戏移动端 H5 化与题材重构综合战略报告

## 1. 执行摘要与战略愿景

本报告旨在为开源项目 `react-xiuxian-game` 的深度迭代提供一份详尽的战略规划、设计文档与技术实施指南。针对阁下提出的将现有“修仙文字RPG”转型为“竖屏H5移动端游戏”，并进行“辐射（Fallout）废土题材”换皮的核心需求，本方案将详细阐述如何在**保持原有数值体系与底层系统不变**的前提下，通过叙事重构、视觉包装与前端技术升级，实现产品的彻底转型与西方市场的本地化适配。

### 1.1 项目背景与核心挑战

原项目 `react-xiuxian-game` 是一个基于 React 19、TypeScript 和 Vite 构建的修仙题材放置类游戏，其核心玩法循环为“挂机修炼（资源积累）→ 突破境界（数值验证）→ 历练/战斗（RNG事件与战利品获取）”1。

本次迭代面临三大核心挑战：

1. **题材跨越的认知失调**：如何将东方玄幻的“灵气”、“渡劫”、“元婴”等概念，无缝映射到西方“废土”、“辐射”、“变异”的语境中，且不破坏原有数值逻辑。
2. **移动端体验重构**：原项目为桌面端设计，需彻底重构为适应单手操作的竖屏 H5 移动应用（PWA），并解决移动端浏览器的视口适配与交互问题。
3. **全球化部署**：利用 Cloudflare Pages 实现高性能的全球分发与持续集成。

### 1.2 战略定位：从“修真”到“生存”

在西方游戏市场，尤其是 RPG 领域，*Bethesda* 的《辐射》系列构建了极具辨识度的“后启示录”美学与生存逻辑。分析表明，修仙游戏的“逆天改命”内核与废土题材的“末世求生”内核在底层机制上具有高度的同构性：

- **修炼** $\Leftrightarrow$ **生存/变异适应**：积累能量以强化肉体。
- **境界** $\Leftrightarrow$ **基因锁/进化阶段**：生命层次的跃迁。
- **灵石** $\Leftrightarrow$ **瓶盖（Caps）**：硬通货资源。
- **宗门** $\Leftrightarrow$ **派系（Factions）**：钢铁兄弟会、英克雷等组织结构。

本报告将基于这一同构性，制定详细的“换皮”字典与开发规范，确保产品既能保留原有的数值乐趣，又能精准击中西方玩家对“复古未来主义（Retro-futurism）”的审美偏好。

## 2. 策划案：废土题材重构与游戏设计 (GDD)

### 2.1 世界观重构：废土编年史

**游戏名称（暂定）：** *Wasteland Ascendant: Pip-Boy Edition* (废土飞升：哔哔小子版)

核心叙事：

玩家不再是一名在大千世界寻求长生的修士，而是一名从战前避难所（Vault）苏醒的“居民”。外面的世界已被核战争摧毁，充斥着辐射、变异生物与疯狂的掠夺者。玩家的目标是通过收集废土上的资源（原灵气），改造自身的基因序列（原修炼），植入赛博义体（原法宝），最终进化为能够主宰废土的“完美人类”或“超凡变异体”。

界面隐喻：

整个游戏 UI 将被包装为《辐射》系列中经典的 Pip-Boy 3000（个人信息处理器）界面。玩家不是在“玩游戏”，而是在操作手腕上的终端机来管理自己的生存状态 2。

### 2.2 数值系统与术语映射 (The Great Mapping)

这是本次迭代最核心的策划工作。我们需要建立一套严格的映射表，将修仙术语转化为废土术语，同时保持数值不变。

**突破机制 (Breakthrough) 的重新包装：**

- **原机制**：积攒修为 $\rightarrow$ 尝试突破 $\rightarrow$ 成功则升级/失败则扣除修为和血量。

- 新机制

  ：

  基因编辑手术 (Gene Splicing Surgery)

  。

  - 玩家在积攒足够的“基因数据（修为）”后，使用“自动医疗舱（Auto-Doc）”进行手术。
  - **失败惩罚**：手术失败导致“基因崩溃”或“排异反应”，扣除大量生命值并损失部分基因数据（修为）。
  - **渡劫（雷劫）**：**辐射风暴 (Rad Storm) / 病毒排异 (Viral Rejection)**。突破大境界时，玩家必须在剧烈的辐射风暴或FEV病毒改造中存活下来 9。

## 3. 需求文档 (PRD)：移动端垂直化与 H5 体验

### 3.1 移动端交互设计 (Mobile-First UX)

### 3.1.1 视口与布局控制

针对 H5 手游化，必须严格控制浏览器的视口行为，禁止缩放，提供原生 App 般的体验。

功能需求 (FR-01)：

应用必须在 <head> 中包含以下 meta 标签配置，以防止移动端浏览器的双击缩放和手势缩放，并锁定竖屏显示 17：

HTML

<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no, viewport-fit=cover"> <meta name="apple-mobile-web-app-capable" content="yes"> <meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">

### 3.1.2 竖屏 UI 布局规范

- **布局逻辑**：采用 **Flexbox** 或 **Grid** 布局，主容器宽度锁定为 `100%`，但在桌面端调试时应限制 `max-width: 480px` 并居中，模拟手机屏幕。

- 导航栏 (Bottom Navigation)

  ：将原有的顶部或侧边菜单重构为底部的 

  Tab Bar

  （标签栏），包含 4-5 个核心入口：

  1. **Status (状态)**：S.P.E.C.I.A.L. 属性面板与 Pip-Boy 小人状态图。
  2. **Inv (物品)**：物品栏与装备界面。
  3. **Radio (历练)**：挂机日志与随机事件文字流（模拟电台广播）。
  4. **Data (宗门/任务)**：派系任务与世界地图。
  5. **Map (地图/副本)**：探索界面。

- **触控优化**：所有交互按钮（Button）的点击热区不得小于 **44x44 CSS pixels**，以符合手指点击标准 20。

### 3.1.3 视觉风格：Pip-Boy 绿色荧光屏

- 配色方案

  ：全局使用单色 CRT 风格。

  - **背景色**：`#0a0f0a` (极深绿/黑)
  - **主色调**：`#1aff1a` (Pip-Boy 荧光绿)
  - **警告色**：`#ffb642` (琥珀色，用于低血量或辐射警告)
  - **高亮色**：`#e6ffe6` (高亮文本)

- **字体**：必须引入 Web Font 实现复古终端效果。推荐使用开源字体 **Monofonto** 或 **VT323** 22。

- **特效**：CSS 实现 CRT 扫描线（Scanlines）与屏幕微弧度畸变效果，增强沉浸感 24。

### 3.2 国际化需求 (i18n)

- **默认语言**：英语 (en-US)。
- **技术实现**：使用 `react-i18next`。必须将所有硬编码的中文文本（如“修为”、“突破”）提取到 JSON 语言包中。
- **文本量预估**：约 2000-3000 词条（含物品描述、UI 标签、事件文本）。

### 4.2.2 国际化 (i18n) 架构设计

为了支持中英文切换并重点优化英文体验，采用 `public/locales` 目录结构管理翻译文件 30。建议**不要**将翻译文件放入 `src`，而是放入 `public` 以便支持 HttpBackend 异步加载（虽然对于小型游戏，打包进 bundle 也可以，但在 Vite 中推荐使用动态导入或 public 资源）。

**目录结构：**

```
/public   /locales     /en       common.json    // 通用UI：确认、取消、设置       stats.json     // 属性名：HP, Rads, Firepower       items.json     // 物品名与描述       events.json    // 随机事件文本       ranks.json     // 境界/军衔名称     /zh      ...
```

**配置代码 (`src/i18n.ts`)：**

TypeScript

`import i18n from 'i18next'; import { initReactI18next } from 'react-i18next'; import Backend from 'i18next-http-backend'; import LanguageDetector from 'i18next-browser-languagedetector';

i18n .use(Backend) .use(LanguageDetector) .use(initReactI18next) .init({ fallbackLng: 'en', debug: import.meta.env.DEV, interpolation: { escapeValue: false, }, backend: { loadPath: '/locales/{{lng}}/{{ns}}.json', }, defaultNS: 'common', ns: ['common', 'stats', 'items', 'events', 'ranks'], });

export default i18n;`

### 4.3 Cloudflare Pages 部署流水线

Cloudflare Pages 提供了极佳的静态站点托管服务，不仅免费且速度快，非常适合此类纯前端项目 33。

**部署步骤：**

1. 准备 GitHub 仓库：

   确保代码已推送到 GitHub，且包含 vite.config.ts 和 package.json。

2. **Cloudflare Dashboard 操作**：

   - 登录 Cloudflare，进入 **Workers & Pages**。
   - 点击 **Create Application** -> **Pages** -> **Connect to Git**。
   - 选择你的仓库 `react-xiuxian-game`。

3. **构建配置 (Build Settings)**：

   - **Framework Preset**: 选择 `Vite`。

   - **Build Command**: `npm run build` (或 `pnpm build`)。

   - **Build Output Directory**: `dist`。

   - Environment Variables

     :

     - 设置 `NODE_VERSION` 为 `18` 或更高，确保兼容 Vite 6。

4. SPA 路由重定向配置：

   由于是单页应用（SPA），需要处理路由刷新 404 问题。在 public 目录下新建一个名为 _redirects 的文件，内容如下 20（注：Cloudflare Pages 默认支持 SPA，通常只需指定 index.html 为入口，但在某些高级路由下，显式规则更安全）：

   /* /index.html 200

   这告诉 Cloudflare 将所有请求都返回 index.html，交由 React Router 处理。
